# Архив кода — REST API для навигации по кодовой базе

Сервис, который индексирует Python-файлы и отдаёт их структуру через REST API. Индексатор на стандартном модуле `ast` извлекает из `.py`-файлов функции и классы (имя, диапазон строк, docstring) и складывает их в SQLite. Приложение на FastAPI отдаёт эти данные в JSON через четыре эндпоинта. Swagger-документация доступна автоматически на `/docs`.

Кейс 2 «Архив кода». Стек: Python 3.12, FastAPI + uvicorn, SQLAlchemy 2.x (ORM поверх SQLite), стандартный модуль `ast`.

## Быстрый старт

```bash
# 1. Установить зависимости
pip install -r requirements.txt

# 2. Проиндексировать датасет в SQLite (по умолчанию папка dataset/)
python -m app.indexer dataset/

# 3. Запустить API
uvicorn app.main:app --reload
```

После запуска:

- Swagger UI: <http://127.0.0.1:8000/docs>
- Пример: <http://127.0.0.1:8000/api/files>

API стартует даже на пустой/неиндексированной базе (таблицы создаются на старте) и возвращает пустые массивы вместо ошибок.

## Схема базы данных

Две таблицы: `files` (один проиндексированный файл) и `definitions` (функции и классы, многие-к-одному к файлу).

### Таблица `files`

| Поле | Тип | Ограничения | Назначение |
| --- | --- | --- | --- |
| `id` | INTEGER | PRIMARY KEY | Первичный ключ |
| `name` | STRING | UNIQUE, INDEX, NOT NULL | Имя файла |
| `path` | STRING | NOT NULL | Относительный путь до файла |

### Таблица `definitions`

| Поле | Тип | Ограничения | Назначение |
| --- | --- | --- | --- |
| `id` | INTEGER | PRIMARY KEY | Первичный ключ |
| `file_id` | INTEGER | FK → files.id, INDEX, NOT NULL | Ссылка на файл |
| `name` | STRING | INDEX, NOT NULL | Имя функции или класса |
| `kind` | STRING(16) | NOT NULL | Тип: `function` или `class` |
| `line_start` | INTEGER | NOT NULL | Строка начала определения |
| `line_end` | INTEGER | NOT NULL | Строка конца определения |
| `docstring` | TEXT | NULL | Docstring, если есть |

Связь: `File 1 — * Definition` (`cascade="all, delete-orphan"`: удаление файла удаляет его определения).

```
files                          definitions
┌────────────┐                 ┌──────────────┐
│ id (PK)    │ 1 ───────────── *│ file_id (FK) │
│ name (U,I) │                  │ id (PK)      │
│ path       │                  │ name (I)     │
└────────────┘                  │ kind         │
                                │ line_start   │
                                │ line_end     │
                                │ docstring    │
                                └──────────────┘
```

## Обоснование решений

Это самая частая тема вопросов на защите — поэтому фиксируем явно.

- **Нормализация на две таблицы.** Имя и путь файла хранятся один раз в `files`, а не дублируются в каждой строке определения. Связь по `file_id`.
- **`kind` строкой (`function`/`class`), а не отдельными таблицами.** Двух типов достаточно; это проще и напрямую обслуживает фильтр `?type=` без `UNION` или join'ов между таблицами.
- **Индексы поставлены под конкретные запросы:**
  - `files.name` (UNIQUE + INDEX) — ускоряет `GET /api/files/{name}/structure` (поиск файла по имени) и гарантирует уникальность имени.
  - `definitions.name` (INDEX) — ускоряет `GET /api/search` по имени.
  - `definitions.file_id` (INDEX) — ускоряет join и выборку структуры файла.
- **Поиск через `LIKE` (а не FTS).** Для датасета ~60 файлов полнотекстовый индекс избыточен; регистронезависимый `LIKE` по `lower(name)`/`lower(docstring)` проще и достаточен.
- **SQLAlchemy ORM вместо чистого `sqlite3`.** Хотелось показать работу с ORM (модели, связи, сессии); для трёх-четырёх эндпоинтов накладные расходы незначительны.
- **Одинаковые имена файлов из разных папок.** `files.name` уникально, поэтому при коллизии индексатор откатывается на относительный путь (`tests/utils.py` вместо второго `utils.py`), сохраняя обе записи различимыми.

## Эндпоинты API

| Метод | Эндпоинт | Описание |
| --- | --- | --- |
| GET | `/api/files` | Список всех файлов с количеством функций в каждом. Пагинация `?limit=&offset=`. |
| GET | `/api/files/{name}/structure` | Полная структура файла: функции и классы со строками и docstring. |
| GET | `/api/search?q={keyword}` | Поиск функций/классов по имени или docstring (регистронезависимо). Фильтр `?type=function\|class`, пагинация `?limit=&offset=`. |
| GET | `/api/stats` | Сводная статистика: количество файлов, функций, классов и определений. |

### Поведение и коды ответов

- Пустой результат у списочных эндпоинтов → пустой массив `[]` со статусом **200** (никогда не 500).
- Файл без определений → `definitions: []`.
- Несуществующий файл в `/structure` → **404**.
- Недопустимое значение `?type=` → **422** (валидация).

## Примеры запросов

```bash
# Все файлы
curl http://127.0.0.1:8000/api/files

# Постранично
curl "http://127.0.0.1:8000/api/files?limit=5&offset=0"

# Структура конкретного файла
curl http://127.0.0.1:8000/api/files/auth.py/structure

# Поиск по ключевому слову
curl "http://127.0.0.1:8000/api/search?q=auth"

# Поиск только классов
curl "http://127.0.0.1:8000/api/search?q=user&type=class"

# Поиск с пагинацией
curl "http://127.0.0.1:8000/api/search?q=user&limit=10&offset=0"

# Сводная статистика
curl http://127.0.0.1:8000/api/stats
```

Пример ответа `GET /api/files`:

```json
[
  {"id": 1, "name": "auth.py", "path": "src/auth.py", "function_count": 3}
]
```

Пример ответа `GET /api/search?q=token`:

```json
[
  {
    "id": 1,
    "name": "TokenService",
    "kind": "class",
    "line_start": 4,
    "line_end": 14,
    "docstring": "Issue and validate tokens.",
    "file_name": "auth.py",
    "file_path": "src/auth.py"
  }
]
```

Пример ответа `GET /api/stats`:

```json
{"files": 6, "functions": 12, "classes": 4, "definitions": 16}
```

## Индексатор

CLI-скрипт, обходящий директорию рекурсивно (`*.py`), парсящий каждый файл через `ast` и сохраняющий функции, классы, async-функции, методы и вложенные определения.

```bash
# По умолчанию: источник dataset/, база code_index.db в корне проекта
python -m app.indexer dataset/

# Свой путь к базе
python -m app.indexer dataset/ --db dataset/code_index.db

# Уровень логирования
python -m app.indexer dataset/ --log-level DEBUG
```

Особенности:

- **Идемпотентность** — перед каждым прогоном таблицы очищаются, повторный запуск не плодит дубли.
- **Устойчивость** — битый файл (синтаксическая ошибка, проблема кодировки) логируется и пропускается, прогон не падает.
- **Файлы читаются как UTF-8**, `end_lineno` берётся из `ast` (Python 3.8+).

Путь к базе можно задать и через переменную окружения `ARCHIVE_DATABASE_URL`.

## Тесты

```bash
pytest
```

Покрытие (20 тестов):

- **Индексатор** — извлечение функций/классов/async/методов/вложенных, номера строк и docstring, пропуск битого файла, уникальность одинаковых имён файлов, идемпотентность повторной индексации.
- **API** — happy-path по каждому эндпоинту и ключевые граничные случаи: пустой результат → `[]` (а не 500), файл без определений → `[]`, несуществующий файл → 404, фильтр `?type=`, пагинация, `/api/stats` на пустой базе.

Каждый тест работает со своей временной SQLite-базой, реальный `code_index.db` не затрагивается.

## Docker

```bash
# Сборка
docker build -t code-archive .

# Запуск (индексирует dataset/ на старте, затем поднимает API на :8000)
docker run -p 8000:8000 code-archive

# Свой датасет вместо запечённого в образ
docker run -p 8000:8000 -v "$(pwd)/dataset:/app/dataset" code-archive
```

Swagger будет на <http://127.0.0.1:8000/docs>.
