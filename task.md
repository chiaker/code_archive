# Кейс 2 -«Архив кода»

_REST API для навигации по кодовой базе: индексатор + три эндпоинта на FastAPI_

## О чём кейс

Первый день в новой IT-компании. Перед вами сотни тысяч строк кода в десятках файлов. Вы хотите задать простой вопрос: «где здесь обрабатывается авторизация?» -но поиск ответа занимает часы. Старшие коллеги тратят время на устные объяснения вместо своей работы, а вы чувствуете себя потерянным.

Проблема решается, если у компании есть структурированный каталог кода -сервис, который знает, какие функции и классы есть в каждом файле, и умеет отдавать эту информацию через API. Любой новичок может задать вопрос HTTP-запросом и получить структурированный JSON-ответ.

Задача команды -написать такой сервис. На Python, с REST API, с базой данных. Никакого Docker, никакой Java и Spring Boot -только классический минималистичный стек, на котором делают подобные инструменты в реальности.

## Что предстоит сделать

Сервис состоит из двух взаимосвязанных частей. Обе пишутся на Python.

**Часть А. Индексатор кода**

Python-скрипт, который читает набор .py-файлов из указанной директории и извлекает из них структурную информацию: имена файлов, имена функций и классов, номера строк начала и конца определения, docstring (если он есть). Вся эта информация сохраняется в файл базы данных SQLite по схеме, которую команда проектирует сама.

Главный инструмент здесь -стандартный модуль ast из Python. Он умеет парсить Python-код в дерево синтаксических узлов, и из этого дерева легко достать всё нужное: функции, классы, их имена, строки, документацию.

**_Подсказка:_** _Базовый пример: tree = ast.parse(open(filepath).read()); for node in ast.walk(tree): if isinstance(node, ast.FunctionDef): print(node.name, node.lineno). Никаких сторонних библиотек для парсинга устанавливать не нужно -модуль ast встроен в Python._

**Часть Б. REST API на FastAPI**

Веб-приложение на FastAPI, которое предоставляет три эндпоинта для работы с проиндексированными данными. Все эндпоинты возвращают ответы в формате JSON. Если по запросу ничего не найдено, эндпоинт должен вернуть пустой массив (а не ошибку 500). Это важное требование -типичная ошибка студентов: вернуть 500 при пустом результате.

| **Эндпоинт**                    | **Что делает**                                                                                                |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| GET /api/files                  | Возвращает список всех проиндексированных файлов с количеством функций в каждом                               |
| GET /api/files/{name}/structure | Возвращает полную структуру конкретного файла: все функции и классы с номерами строк и docstring              |
| GET /api/search?q={keyword}     | Возвращает все функции и классы, в имени или описании которых встречается ключевое слово (регистронезависимо) |

**_Подсказка:_** _FastAPI -это микро-фреймворк, на котором сервис из трёх эндпоинтов пишется за час. Старт: pip install fastapi uvicorn → app = FastAPI() → @app.get("/api/files") def list_files(): return \[...\] → uvicorn main:app --reload. Swagger-документация появляется автоматически по адресу /docs -её удобно показывать на защите._

## Какие навыки потребуются

| **Навык**         | **Уровень**  | **Где применяется**                                |
| ----------------- | ------------ | -------------------------------------------------- |
| Python            | Уверенный    | Основной язык всего проекта                        |
| HTTP              | Базовый      | Методы GET/POST, статус-коды 200/404/500           |
| REST API          | Базовый      | Что значит /api/users/{id}, как устроены endpoints |
| SQL               | Базовый      | SELECT, WHERE, JOIN, простые индексы               |
| SQLite            | Базовый      | Соединение, запрос, курсор                         |
| SQLAlchemy (опц.) | Базовый      | Можно вместо чистого sqlite3 -на выбор             |
| FastAPI           | Базовый      | Три эндпоинта, запуск локально                     |
| Postman или curl  | Базовый      | Проверка API без браузера                          |
| ast (модуль)      | По туториалу | Парсинг Python-файлов на функции/классы            |
| JSON              | Базовый      | Формат ответов API                                 |

**_Подсказка:_** _Если выбираете между чистым sqlite3 и SQLAlchemy -для трёх эндпоинтов чистый sqlite3 проще и нагляднее. SQLAlchemy имеет смысл, если хочется попробовать ORM и не страшно потратить пару часов на изучение. Эксперты не штрафуют за «упрощённый» выбор, если он обоснован в README._

## Что НЕ нужно делать

Кейс намеренно ограничен по сложности. Эти задачи в данном кейсе не решаются:

- Не нужно делать Docker и docker-compose (можно как бонус, но не обязательно)
- Не нужно деплоить на сервер или в облако
- Не нужно знать продвинутые ORM-фичи (миграции через Alembic, ленивая загрузка, eager loading)
- Не нужно оптимизировать запросы (планы выполнения, бенчмарки)
- Не нужно делать фронтенд -взаимодействие только через API
- Не нужно решать вопросы безопасности (аутентификация, rate limiting)

## Дополнительные возможности (плюс к оценке)

Эти задачи не обязательны, но добавляют баллов на защите:

- Фильтрация поиска по типу: GET /api/search?q=user&type=class или ?type=function
- Эндпоинт GET /api/stats со сводной статистикой (сколько файлов, функций, классов в БД)
- Пагинация на списочных эндпоинтах: ?limit=10&offset=20
- Dockerfile и одна команда docker build + docker run -для тех, кто знаком с Docker

## Результат сдачи

Команда публикует код в открытом репозитории GitHub и проводит защиту в ВКС (7 минут): запускает сервис, индексирует датасет на 60 предоставленных .py-файлов и демонстрирует все три эндпоинта запросами, которые эксперты задают вживую.

**_Подсказка:_** _На защите эксперты почти наверняка спросят про устройство базы данных: какая схема, почему именно такая, какие индексы. Это самая частая тема вопросов. Хороший ответ начинается со слов: «я выбрал такую схему, потому что...» и заканчивается обоснованным аргументом. Простое «так получилось» -почти гарантированный минус балл._

# Общие требования к обоим кейсам

## Состав команды

От 2 до 4 участников. Распределение ролей -на усмотрение команды, но опыт показывает, что для отборочного тура работают две схемы. Первая -все вместе пишут код и комментируют друг другу. Вторая -один отвечает за «инфраструктуру» (запуск, README, репозиторий), второй за основную логику, остальные за тесты и презентацию. Обе схемы валидны.

## Стек технологий

Кейсы намеренно собраны на минималистичном стеке: Python и его стандартные библиотеки плюс одна-две внешних библиотеки на каждый кейс. Никакой Java, Spring Boot, PostgreSQL, React или Kubernetes -для отборочного тура это избыточно.

| **Кейс** | **Основной стек**                                                                        |
| -------- | ---------------------------------------------------------------------------------------- |
| Кейс 1   | Python 3.12, Jupyter Notebook, sentence-transformers, numpy, pandas, matplotlib, sklearn |
| Кейс 2   | Python 3.12, FastAPI, uvicorn, SQLite (sqlite3 или SQLAlchemy), модуль ast               |

## Защита

Защита проходит в ВКС, продолжительность 7 минут на команду: 5 минут на демонстрацию и устный рассказ, 2 минуты на вопросы. Программу демонстрации команда определяет сама, но эксперты ждут конкретики -таблица результатов, живые запросы к API, понятный финальный вывод.

## Критерии оценки (общие принципы)

- Корректность результата -главный фактор. Метрики посчитаны верно или API работает без ошибок
- Обоснованность решений -почему именно этот стек, эта схема БД, эта модель
- Воспроизводимость -проект запускается из чистой среды одной командой после pip install -r requirements.txt
- Документация -README с описанием схемы (для Кейса 2) или инструкцией к ноутбуку (для Кейса 1)
- Качество защиты -понятная презентация, готовность отвечать на вопросы

## Что делать, если возникли вопросы

Вопросы по кейсам присылайте организаторам через официальный канал чемпионата. Если вопрос касается формата данных, неоднозначности в формулировке или технической детали -отвечаем в течение рабочего дня. Если вопрос касается «как лучше реализовать X» -на такие вопросы не отвечаем, это часть оценки.
